System Design: Scaling

How to support our system to zero to millions of users? I will discuss it here step by step. It’s common that we start something simple and change when demand grows. In Fig: 1, our system is simple initially where users are connected with the server. Here the web server, database and cache – everything running under one server.

Fig: 1


So, what is the problem with the design? As the web application will be more popular, the server will not be able to support more user traffic. So, the ideal thing to do in this situation is separating the data tier from the web tier. Here is what the design looks like (Fig: 2).

Fig: 2


Here, the web tier will take care of all the web traffic and the database tier will take care of all the data growth. The advantage is that we can now scale our web tier and data tier independently. Meaning, if lot of users come, we can increase the capacity of our web tier. And, if user is not that high but produces a lot of then we can increase the capacity of data tier without increasing the capacity of web tier.

 

Now the question is, how to increase the capacity? There are to ways of doing that- one is Vertical Scaling and the other is Horizontal Scaling.


Let’s talk about vertical scaling. It means increasing the capacity of the existing server. Meaning Adding more RAMs, Hard Disks or more CPUs. The architecture will be the same but with more added capacity.

What are the pros and cons of this?

Pros:
Simplicity, its relatively simple to add more RAMs and CPUs to an existing system.

Cons:
Impossible to add infinite memory and CPU to one server.
No failover or redundancy. Meaning, if one server goes down, the entire system goes down.


Now let’s discuss Horizontal Scaling. In Horizontal Scaling, we will add more servers into our pool. Meaning if we have one server, we will add a second server. Likewise, we can add more servers when demand grows. For this, we will also need a load balancer which will distribute loads across the servers based on their health status (See Fig: 3).

Fig: 3


Now, besides balancing loads, the system is also fault tolerant. Meaning, if Server 1 goes down, the application still runs using Server 2. Also look that we have added two read replica server in the database tier. The purpose of this is that, they will provide all read operations for the web servers. And the master database will perform all insert/ update operations for the web servers. It will also keep data sync in replica sets. In addition, if the master database fails, one of the read replica will become master database to server write operations as well.
